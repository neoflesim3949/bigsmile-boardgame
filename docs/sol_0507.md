# 備援方案：雙 Vercel + 雙 Supabase 可行性分析

> 撰寫日期：2026-05-07
> 來源：用戶問「可以同時在兩組 vercel 跟 supabase 跑備援嗎？」
> 結論：**冷備援可行且推薦**；熱備援（雙活）對一場 2 小時活動 CP 值太低，不建議

---

## TL;DR

| 方案 | 可行性 | 成本（時間） | 切換時體驗 | 推薦 |
|------|-------|-----------|----------|------|
| **冷備援**（standby DB 平時不接流量）| 🟢 高 | 1–2 小時建置 | 切換瞬間玩家重登、丟最後幾分鐘 | ✅ **推薦** |
| **熱備援**（雙活、雙寫）| 🔴 極低（架構性不可行）| 數天～數週 | 一致性風險高 | ❌ 不建議 |
| 不做備援、純 Supabase snapshot | 🟢 高 | 0 | 主站掛 = 活動中止 | 🟡 最低保險 |

---

## 1. 核心矛盾：為何雙活（熱備援）不可行？

### 1.1 問題本質

兩組 Supabase = 兩個獨立 Postgres DB。要「同時跑」表示：
- 玩家 A 從 Vercel 站 #1 登入 → 寫入 Supabase #1
- 玩家 B 從 Vercel 站 #2 登入 → 寫入 Supabase #2

→ **狀態分裂**：A 在 #1 有 1000 元、在 #2 帳號根本不存在；A 跟 B 互相轉帳會在不同 DB 上各扣各的錢，對不上。

### 1.2 要解決必須引入

| 工具 | 何種用途 | 對本系統的代價 |
|------|---------|--------------|
| **Postgres logical replication** | 兩 DB 互推 WAL 變動 | 衝突解決需自己寫（同 user_id 同時被改怎辦？）；schema 需符合 replicate-friendly（無 sequence 衝突等）|
| **PgEdge / Citus / Yugabyte** | Multi-master 分散式 PG | 換掉 Supabase（=放棄 PostgREST / Realtime / Auth 整套生態）|
| **應用層雙寫** | server action 同時寫 A/B | 任一寫失敗的 rollback 邏輯；延遲翻倍；交易隔離不再原子化 |

→ 為一場 2 小時活動投資數天～數週工程，**回報極低**。

### 1.3 結論

**雙 Supabase 同時收寫入 = 為自己造一個比 Supabase 掛掉更可能炸的 bug 源**。

---

## 2. 推薦：冷備援（Cold Standby）

### 2.1 架構

```
[玩家] ──DNS──> bigsmile-boardgame.vercel.app
                    │
                    ▼ (env var 指向)
                supabase-A.co  ←  主站（活動進行中唯一寫入點）
                    │
                    ▼ (定期 snapshot dump)
                supabase-B.co  ←  備援站（平時不接流量、只接快照還原）
                                    │
                                    ▼ (緊急時手動切)
                                  bigsmile-boardgame-failover.vercel.app
```

### 2.2 平時準備（活動前一天）

1. **Supabase 建第二個 project**（命名 `bigsmile-boardgame-standby`、同 region `ap-northeast-1`）
2. 在 standby 跑完整 migration：`DATABASE_URL=postgresql://...standby... npm run db:migrate`
3. 在 standby 跑 `db:seed:prod`（建 admin 帳號、預設股票、命格範本等）
4. **Vercel 建第二個 project**（從同一個 GitHub repo deploy、命名 `bigsmile-boardgame-failover`）
5. 第二個 Vercel project 的環境變數指向 standby Supabase
6. 確認 failover 站可正常登入（用 admin 測一次）

### 2.3 活動進行中的同步策略

| 策略 | 同步間隔 | 切換時資料丟失 | 實作工 |
|------|---------|------------|--------|
| **手動 snapshot**（活動前 + 中場）| 60 分鐘 | 最多 60 min | ⭐ 0（Supabase Dashboard 點兩下）|
| **定時 pg_dump cron** | 5–10 分鐘 | 5–10 min | 需寫 cron + 上傳到 standby（中等）|
| **logical replication** | 即時 | < 1 min | 高（要設 publication / subscription / 衝突處理）|

→ 對 2 小時活動，**手動 snapshot 中場一次**已經足夠（最壞情況丟 60 分鐘狀態 = 半場活動，可重啟）。

### 2.4 緊急切換 Runbook（活動進行中主站掛了）

**前提**：已有最近的 standby snapshot；DNS 或 Vercel domain 可即時改。

```
1. 大會廣播：「系統暫停 5 分鐘，請玩家停在原地」
2. （Supabase 主站）拉最後一份 snapshot → 還原到 standby
   - 若主站完全失聯，跳過此步，直接用上一次 snapshot
3. （Vercel）切換 production domain：
   方法 A：把 bigsmile-boardgame.vercel.app 的 alias 從 main project 移到 failover project
   方法 B：改 DNS（如果用自訂 domain）
4. 通知玩家：「請重新登入」（JWT token 對 standby 是無效的，必須重登）
5. 大會開放繼續活動
```

預估切換時間：**3–5 分鐘**（步驟 2 的 snapshot restore 是主要瓶頸）。

### 2.5 風險與限制

| 風險 | 影響 | 緩解 |
|------|-----|------|
| Snapshot 落後 60 分鐘 | 玩家最後一回合的金錢 / 持股 / 道具丟失 | 中場提醒「狀態以系統為準」、活動 SOP 寫明 |
| JWT 跨站不通用 | 切換後所有玩家被踢登入頁 | 接受、廣播提醒 |
| 兩組 Supabase 雙倍費用 | Pro tier $25 × 2 = $50/月 | 用免費 tier 兩組（每個 project 都 < 500 row 沒問題）|
| 切換需人工操作 | 主辦人忙中容易切錯 | Runbook 印紙本、事前演練 1 次 |

## 3. 最低保險方案：純 Supabase Snapshot（無備援站）

### 3.1 適用情境
- 預算 / 時間極緊
- 主辦方接受「主站掛了 = 活動中止 + 改天重辦」

### 3.2 步驟
1. 活動前 1 小時：Supabase Dashboard → Database → Backups → Create backup
2. 活動中場（60 分鐘）：再建一次
3. 主站若崩 → 活動中止、事後 restore 不影響後續活動（資料保住）

### 3.3 缺點
- 沒有「活動進行中切換到備援站繼續玩」的能力
- 但對「萬一炸了不至於資料全失」這個目標來說足夠

## 4. 工具腳本：`scripts/setup-standby.ts`（可選）

若要做冷備援，需要把整套 migration + seed 套到 standby DB。建議封裝：

```ts
// scripts/setup-standby.ts
// 用法：STANDBY_DATABASE_URL=postgresql://... npx tsx scripts/setup-standby.ts
import { config as loadEnv } from 'dotenv';
loadEnv({ path: '.env.local' });

const standbyUrl = process.env.STANDBY_DATABASE_URL;
if (!standbyUrl) {
  console.error('STANDBY_DATABASE_URL not set');
  process.exit(1);
}

// 把 DATABASE_URL 暫時 override 為 standby、跑 migrate + seed:prod
process.env.DATABASE_URL = standbyUrl;
await import('./migrate.ts');
await import('./seed-prod.ts');
console.log('✓ Standby ready');
```

執行：
```bash
STANDBY_DATABASE_URL="postgresql://...standby..." npx tsx scripts/setup-standby.ts
```

→ 一條指令搞定 standby 的 schema + 預設資料。

## 5. 成本對照（2 小時活動的單次成本）

| 方案 | Supabase | Vercel | 工程時間 | 切換時體驗 |
|------|----------|--------|---------|----------|
| 不做備援 + snapshot | 免費（單一 project）| 免費 | 0 | 活動中止 |
| **冷備援**（snapshot 中場 1 次）| 免費 ×2 project | 免費 ×2 project | 1–2h 建置 | 重登 + 丟 ≤ 60 min |
| 冷備援 + 5 min cron sync | 免費 ×2 + cron VM ($5/月) | 免費 ×2 | 4–6h | 重登 + 丟 ≤ 5 min |
| 熱備援（不可行）| Pro+ 費用、replication 工具 | — | 數天～數週 | 一致性 bug |

---

## 6. 結論

對「**單場 2 小時、≤ 500 人**」的活動：

- ✅ **建議冷備援**：花 1–2 小時建第二組 Vercel + Supabase、活動中場手動 snapshot 一次。投資報酬率最高
- ✅ **底線保險**：什麼都不做也要在活動前 + 中場各做一次 Supabase snapshot
- ❌ **不要做雙活**：雙寫 / 多 master 對這規模是 over-engineering，引入的 bug 比要救的故障還多

### 行動清單（若要做冷備援）

- [ ] Supabase 建 standby project
- [ ] 在 standby 跑 migration + seed:prod（或寫 `scripts/setup-standby.ts`）
- [ ] Vercel 建 failover project、env 指向 standby
- [ ] 用 admin 帳號到 failover 站測一輪登入 / 推進回合 / 套快捷模組
- [ ] 印一份切換 Runbook（§2.4）給主辦人
- [ ] 活動前 1 小時 + 中場 60 分鐘 snapshot
