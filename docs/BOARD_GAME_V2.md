## 1. 系統概述

開運大富翁是一個**獨立的活動型小遊戲系統**，可單獨部署、自帶帳號與權限管理。由大會管理員啟用活動後，玩家以帳號登入即可進入遊戲；管理端、關主端、玩家端各自擁有獨立的入口與介面。

系統包含四項核心遊戲參數（金錢、健康、福分、業力）、可由關主發放的道具、以及內建的小型股市。所有寫入皆走後端伺服器並產生交易紀錄，作為排行榜、稽核與爭議處理依據。

## 2. 頁面架構

> **路由詳細清單以架構文件 §4 為單一正本**；本節僅列玩家視角的主要頁面分類。

| 頁面 | 對象 | 主要功能 |
|------|------|----------|
| 大會後台 `/admin/*` | 大會管理員 | 排行榜、回合控制、帳號 / 關卡 / 商品 / 道具 / 事件 / 看板 / 財務 / 參數 CRUD（多個子路由，見 ARCH §4） |
| 關主後台 `/captain` | 關主 | 巨大掃碼入口 + 快捷模組列表入口 |
| 快捷模組編輯 `/captain/actions` | 關主 | 建立 / 修改 / 刪除快捷模組 |
| 關主前台 `/captain/scan` | 關主 | 掃描用戶 QR、進行列表（同時帶多人）、完成後結算 |
| 玩家首頁 `/` | 玩家 | 四項數值卡（點擊進 `/history/[type]`）、道具列表、入口卡、QR 彈窗 |
| 命格抽卡 `/onboarding` | 玩家 | **非**首次登入自動進入。唯一觸發條件：`CardDrawMode=true` AND `destiny_name=NULL` AND 玩家重新整理頁面，三者同時成立時由 middleware 強制導向；其餘情況嘗試進入一律被擋回 `/` |
| 數值明細 `/history/[type]` | 玩家 | 四項數值的完整交易明細（type = money / health / blessing / karma） |
| 換匯所 `/exchange` | 玩家 | 特殊資源 → 金錢兌換（玩家看不到福報資訊） |
| 銀行借貸 `/bank` | 玩家 | 借款方案選擇、還款；利息每回合由系統自動結算 |
| 玩家轉帳 `/transfer` | 玩家 | 輸入 ID 或掃碼查詢收款玩家後匯款 |
| 股市介面 `/stock` | 玩家 | 股票列表、持股庫存、預期利潤、買賣（**不顯示曲線圖**） |
| 玩家偏好 `/settings` | 玩家 | 字級、深色 / 淺色模式（localStorage） |
| 活動看板 `/display/board` | 現場大屏／投影 | 公開展示行情、事件與跑馬燈（**強制深色**，無互動） |

## 3. 大會後台

### 3.1 總覽與儀表板 (`/admin`)
- **即時狀態列**：顯示「遊戲已進行時間」（碼錶形式，例如 01:24:35，從活動點擊開始計算）、「系統狀態」（例如：活動進行中 / 暫停）、「目前回合數」。
  - **防呆提醒機制**：遊戲時間每逢 10 分鐘（例如 00:10:xx, 00:20:xx），該時間欄位會變為紅色並持續閃爍，提醒主控台管理員「該點擊推進下一回合了」。直到管理員點擊「推進下一回合」按鈕後，閃爍才會停止。
- **排行榜**：查看玩家排行榜。顯示欄位包含：排名、姓名、金錢、福份、健康、業力、重生次數、最終分數。點擊數值欄位標題可即時切換排序（Asc / Desc）。**注意：「排名」數字永遠依據「最終分數」高低固定，不會因為切換其他欄位排序而改變玩家的絕對名次。**
- **回合控制面板**（控制台之一）：
  - **推進下一回合**：按下後同時完成「套用本回合股價變化」（依 `StockRoundScript`，無設則 ±5% 隨機 fallback）與「結算所有借款玩家利息（扣金錢與福分、更新時間戳）」，看板自動推進回合數；該回合若有 `StockRoundEvent` 文字會自動推到看板跑馬燈 5 分鐘。**30 秒節流**防誤點。
- **遊戲開關控制**（頂部工具列）：
  - **導覽遊戲** toggle / **抽卡模式** toggle：兩個前置條件，**都開啟後才能按「遊戲開始」**。
  - **遊戲開始**：按下後 `BoardGameEnabled = true`，首次啟動會同時記錄 `BoardGameStartedAt`（碼錶從這時開始計算）。
  - **遊戲結束（最終計分）**：活動結束後按下此鍵，系統將關閉所有玩家操作權限（除 admin），並依設定公式（金錢×Wm + 福分×Wb − 業力×Wk）進行最終結算，結果同時推播至大會活動看板與玩家 App，呈現最終完整排行榜。**不可復原**。
  - **開啟活動看板** link：跳到 `/admin/events` 去發 display token + 開大屏。
- **即時跑馬燈廣播**（控制台之一）：textarea + 發送至看板 + 清除按鈕。可設定自動清除時間（TTL，分鐘），上限由 `BoardMarqueeMaxMinutes` 控制。
- **換匯所即時權重控制**（控制台之一）：
  - 提供 `-50%` / `-20%` / `0%` / `+50%` / `+100%` / 自訂 6 鈕，調整全域 `ExchangeRateMultiplier`（實作範圍 `0–10`，亦即 -100% 到 +900%）。
  - **顯示邏輯**：因為換匯所內可能存在多種不同的「兌換方案」（例如少量兌換與大量兌換的基數不同），總覽面板**不顯示**單一死板的兌換金額，而是純粹顯示當前的「全域倍率」（例如套用 150%）。
  - **計算邏輯**：玩家消耗的福份單位不變，但換得的金錢 = `units × money_gain_per_unit × multiplier`。讓大會能根據現場活動氣氛，迅速對所有方案施加全場「通膨」或「通縮」。

> 「重置股價歷史 / 重置使用次數 / 重置會員明細 / 刪除所有會員 / 刪除所有股票」等不可復原操作，**統一在 `/admin/settings` 的危險操作區**（每個按鈕需 3 次確認），不放在 dashboard 以避免誤觸。

### 3.2 參數設定 (`/admin/settings`)
設定系統全域參數，**6 個 section**：
- **數值顯示設定**：`ShowAllStats` toggle，控制玩家端是否顯示福分／業力（健康／金錢始終可見）。
- **最終計分權重**：最終結算公式的佔比 (`ScoreWeightMoney`, `ScoreWeightBlessing`, `ScoreWeightKarma`)。
- **新手命格範本 (抽卡池)**：可建立多套版本（例如：富貴命、清修命）。玩家的資料表會有一欄 `destiny_name` 紀錄其命格。
  - **玩家手動抽取**：觸發條件**不是**「首次登入」，而是**抽卡模式啟用 + 玩家尚無命格**：當 `AppSettings.CardDrawMode === 'true'` 且玩家 `destiny_name` 為 NULL 時，玩家在重新整理 / 進入頁面時將被 middleware 強制導向「**抽取命格（求籤/抽卡）**」過場畫面。玩家必須親自點擊按鈕進行抽取，系統才會從「已啟用」的命格中隨機分配一套並賦予對應的四項初始參數。抽取完成後才可進入正式遊戲首頁。若 `CardDrawMode === 'false'` 或玩家已有 `destiny_name`，皆略過抽卡頁。
  - 欄位：命格名稱、**emoji（卡面圖示）**、**描述短文（卡面說明）**、**色系**（前端色票枚舉：amber / teal / purple / rose / sky / zinc，控制邊框 / 漸層 / 發光）、**稀有度標籤**（自由輸入字串，例：稀有 / 普通 / 神秘）、四項參數（金錢、健康、福分、業力）、狀態（啟用／停用）。
  - 為什麼色系是枚舉而非自由色碼：Tailwind JIT 必須能在編譯期看到完整的 class 字串才會生成樣式，所以 hex 色碼無法動態套用；改採「色系名稱」的設計讓後台仍可選色、前端有對應的 Tailwind palette。
- **預設新手初始值**：當系統內沒有任何命格範本被啟用時的「防呆備案」，系統將回退使用這組單一預設參數 (`InitialMoney`, `InitialHealth`, `InitialBlessing`, `InitialKarma`) 給新玩家。
- **重生後初始值**：玩家遭關主執行「重生」後的四項初始值 (`RebirthMoney`, `RebirthHealth`, `RebirthBlessing`, `RebirthKarma`)。與新手初始值**分開管理**。
- **危險操作區（Danger Zone）**：5 個按鈕——重置會員明細、刪除所有會員、重置股價歷史、刪除所有股票、重置使用次數。**每個按鈕需經過 3 次確認彈窗才會執行**。

> **此頁不含**：活動時間旗標 / 換匯倍率（在 `/admin` 總覽面板）；換匯方案 / 銀行借貸方案（在 `/admin/finance`）；看板版型 / 跑馬燈 / display token（在 `/admin/events`）。

### 3.3 財務設定 (`/admin/finance`)
管理遊戲內金融系統規則：
- **換匯所方案**：新增／編輯／刪除福報兌換現金的方案。
  - 欄位：方案名稱、每單位消耗福報 (`blessingCost`)、每單位獲得金錢 (`amount`)、描述。
- **銀行借貸規則**：新增／編輯／刪除借款方案。
  - 欄位：方案名稱、借款本金 (`amount`)、每回合扣除福報 % (`blessingRatePerRound`)、每回合利息 % (`interestRatePerRound`)、描述。
  - 介面提供即時預覽，換算 % 數為實際扣除數值。

### 3.4 帳號管理 (`/admin/accounts`)
- **帳號列表**：支援搜尋（姓名／登入帳號）、角色過濾（管理員、關主、玩家）。
- **帳號 CRUD**：新增、編輯姓名與帳號、啟用／停用狀態。
- **登入機制區分**：
  - **玩家**：採用免密碼設計，輸入「**玩家 ID**」+「**姓名**」兩者完全吻合即可登入。
  - **關主與管理員**：維持高安全性，需輸入「**帳號**」+「**密碼**」登入。
    - **初始設置**：由大會管理員在後台建立帳號時，輸入一組初始密碼並告知對方。
    - **自行修改**：關主或管理員登入後，可在各自的後台介面中點擊「個人設定」來自行修改密碼。
- **進階操作**：
  - 重設密碼（若關主忘記密碼，由管理員於此頁面產生隨機臨時密碼覆蓋，關主登入後需再自行修改）
  - 切換角色設定
  - 重設玩家四項參數至初始值（測試或誤發復原）
  - 清空玩家持股與道具
- **批次匯入**：支援上傳 CSV 一次建立多筆帳號。

### 3.5 關卡與關主 (`/admin/stations`)
- **關卡列表**：新增／編輯／刪除實體活動關卡。
  - 欄位：關卡名稱、描述、是否允許該站關主使用重生鍵 (`allow_rebirth`)。
  - 限額設定：每人使用上限 (`player_max_uses`)、全場使用上限 (`global_max_uses`)。
- **關主指派**：為每個關卡指派一位或多位關主（從現有 `captain` 帳號挑選）。

### 3.6 股市商品 (`/admin/stocks`)
- **商品列表**：新增／編輯／刪除股票（上限 10 檔）。
  - 欄位：代碼 (`code`)、名稱 (`name`)、當前價格 (`current_price`)。
  - 屬性開關：前台是否顯示 (`is_visible`)、是否可賣 (`is_sellable`)。
- **回合腳本總表（股市腳本）**：
  - 系統不再單純依賴隨機漲跌，而是提供一個「回合腳本總表」。
  - 管理員可預先設定「第 N 回合」時，各檔股票的漲跌幅度或絕對價格變化。
  - **聯動跑馬燈**：每個回合的股價變化可直接對應一則「事件跑馬燈」（例如：設定第三回合台積電大漲 20%，並綁定跑馬燈公告『AI 晶片需求大爆發，半導體產業全面看漲！』）。
  - 當大會總覽面板按下「下一回合」時，系統會自動讀取腳本總表，套用該回合的股價變動，並同步將對應的事件推播至大會看板的跑馬燈。

### 3.7 道具管理 (`/admin/items`)
- **道具列表**：新增／編輯／刪除由關主發放的道具（如：執照、BUFF 卡）。
  - 欄位：道具名稱、圖示/Emoji (`icon`)、描述、是否啟用 (`is_active`)。

### 3.8 事件與看板 (`/admin/events`)

此頁是**三合一**：劇情事件排程 + 看板畫面設定 + Display Token 管理。

- **劇情事件排程（Event CRUD）**：管理員預先排好的事件，看板底部會依優先度輪播。
  - 欄位：內部標題 (`title`)、顯示文字 (`text`)、開始時間 (`start_at`)、結束時間 (`end_at`)、優先權 (`priority`)、啟用狀態 (`is_active`)。
  - 看板讀取邏輯：`is_active = true` 且當前時間在 `[start_at, end_at]` 區間（NULL 視為不限）；以 `priority DESC, created_at DESC` 排序輪播。
- **看板畫面設定（BoardConfig）**：調整大屏投射的版面與重點關注項目。
  - 主標題文字、重點曲線商品（最多 4 檔）、配色方案（紅漲綠跌 / 綠漲紅跌）、事件輪播間隔秒數。
- **授權 Display Token**：發行或撤銷活動看板專用的 Display Token，供現場大屏授權使用。發行後產生 `https://yourapp.vercel.app/display/board?token=xxx` 連結，可隨時撤銷。

> 「**突發跑馬燈廣播**」（`/admin` 總覽面板的控制台之一）與此頁的「**事件**」是**兩條獨立通道**：跑馬燈即時、單一字串、有 TTL；事件結構化、有排程與優先度。詳見「事件 vs 跑馬燈」對照表。
> 「**股市回合腳本的事件文字**」（`/admin/stocks` 的回合腳本總表）會在 `tickRound` 推進該回合時**自動推到看板跑馬燈** 5 分鐘，不走此頁的事件列。

## 4. 關主後台

### 功能說明
- 設定**快捷功能模組**
- 每個快捷模組可設定：
  - **執行效果**：對玩家發放或收回的數量，作用於四項參數（金錢、健康、福分、業力）
  - **綁定道具**：可選擇性同時發放道具
  - **前提條件**（可選）：玩家必須滿足以下條件才能執行該模組：
    - 四項參數最低門檻（金錢 ≥ N、健康 ≥ N、福分 ≥ N、業力 ≥ N）
    - 必須持有特定道具
    - 條件欄位留空 = 不檢查該項
  - **使用次數限額**（兩種，獨立可空 = 不限）：
    - **每位玩家最多領取 N 次**（`player_max_uses`）— 例：「完成挑戰 +50 金錢」每人最多領 1 次
    - **全活動最多被使用 N 次**（`global_max_uses`）— 例：「中獎 +1000 元」全場只發 5 次
    - 同時受所屬關卡的 `Station.player_max_uses` / `global_max_uses` 限制（任一達上限即拒絕）
  - 條件不符或達上限時，關主點擊該快捷模組會彈窗顯示對應訊息並列出原因
- 快捷模組會綁定至關主前台供操作時使用

## 5. 關主前台

### 操作流程
1. **取得玩家身份**（兩條路徑擇一）：
   - **掃 QR Code**（首選）：用 `/captain/scan` 的相機掃玩家彈出的 QR。
   - **手動輸入 ID**（後備）：在 `/captain/scan` 的「手動輸入玩家 ID」欄位貼上玩家 ID（≥ 6 碼）按查詢。**用途**：相機壞掉、瀏覽器不支援相機（如 LINE 內建瀏覽器）、QR 模糊等情況。
   - **限制**：手動輸入路徑**不會出現重生鍵**，前端 `scanned.source === 'manual'` 時隱藏重生按鈕；後端 `rebirthPlayer` action 仍只收 `qrToken` 參數，無法被繞過（雙重防呆）。重生強制走 QR 掃碼。
2. **顯示用戶資料**：系統顯示該玩家的四項參數（金錢、健康、福分、業力）與持有道具。**不顯示玩家持股資訊**（持股屬於玩家股市視野）。
3. **點擊快捷鍵**：選擇點擊欲套用的「快捷模組」按鈕。
4. **檢查符合前提條件**：系統驗證玩家是否滿足模組規定的最低參數、道具條件及使用次數上限；條件不符彈窗說明。
5. **加入進行列表（執行綁定）**：
   - 條件符合後，關主點擊**「加入進行列表（執行）」**，將玩家與模組「綁定」放入**「進行中列表 (In-Progress List)」**。
   - 此時**尚未正式結算**；關主可繼續掃描下一位玩家，同時帶多人進行活動。
   - 頁面頂部常駐顯示進行中玩家清單（姓名 + 模組名稱），讓關主掌握現況。
6. **點擊完成才發放（正式結算）**：
   - 玩家完成挑戰後，關主在進行列表中對該玩家點擊**「完成結算」**。
   - 系統才正式呼叫後端 `applyQuickAction`：套用四項參數變動、發放道具、計入使用次數、寫入 `Transaction`。
   - 結算後該筆從進行列表消失；若玩家 `health ≤ 0 || blessing ≤ 0` 下次刷新轉為地獄畫面。

## 6. 使用者介面

### 顯示項目
- 姓名
- **金錢**（玩家之間可互轉）、**健康** — 始終顯示，上限 100
- **福分**、**業力** — 僅當 `ShowAllStats = true` 時顯示（或於活動結束時強制全開）。
- **個人歷史明細**：當活動進入「遊戲結束」狀態（大結算）時，這四張數值卡片將轉為「可點擊」的按鈕。玩家點擊單一數值（例如「金錢」），系統便會彈出該項數值在整局遊戲中的「完整交易明細」，讓玩家回顧自己的大起大落。

### 下地獄機制（死亡狀態）
- **觸發條件**：健康歸零（`health ≤ 0`）或福分歸零（`blessing ≤ 0`）
- **狀態表現**：
  - 玩家 App 鎖定為「地獄畫面」，**僅顯示 QR Code + 玩家姓名 + UserID**（旁邊大字）
  - 所有操作停用（換匯、轉帳、股市買賣等皆不可用）
  - 玩家只能展示 QR Code 給關主掃描；若關主相機掃不到，可改用 QR 旁顯示的 ID 由關主**手動輸入查詢**（但手動路徑不會出現重生鍵，仍需設法掃 QR）
- **解除方式**：找到擁有「重生鍵」的關主（`allow_rebirth = true`）掃描 QR Code 並執行重生
  - **重生鍵出現的防呆條件（缺一不可）**：
    1. 該關卡 `Station.allow_rebirth === true`
    2. 玩家當前確實處於地獄狀態（`health ≤ 0 || blessing ≤ 0`）—— 後端 pg 交易內即時讀 `PlayerStats` 二次驗證，避免前端繞過
    3. 玩家**主動**把地獄畫面的 QR Code 拿給關主掃；關主無法手動輸入 ID 或對未死亡玩家任意執行重生
  - 重生後：
    - 四項參數重設為重生初始值。**（重要：系統需計算並寫入一筆「差額」的交易明細。例如重生預設金錢是 5000，玩家死前剩餘 200，則明細會寫入 `+4800`。重生絕對不能清除或抹滅玩家過往的交易紀錄，所有的明細經歷都必須完整保留。）**
    - **持股全數清零**（`StockHolding` 該玩家整列刪除，不折現補償）
    - 銀行借款歸零（`bank_loan = 0`、`loan_updated_at = NULL`）
    - **道具全數清空**（`PlayerItem` 該玩家整列刪除，含手術執照、財神爺 BUFF 等所有身份識別道具）
    - 解除死亡狀態
    - **本質：除了原始抽到的命格 `destiny_name` 外，所有可變狀態歸零重來**
- **判定方式**：不額外儲存死亡旗標，前後端皆以 `health ≤ 0 || blessing ≤ 0` 即時計算

### 功能模組
- **換匯所** `/exchange`：特殊資源 → 金錢兌換
  - **玩家完全看不到福報相關資訊**（餘額、消耗量皆隱藏）。
  - 後台可設定多種「兌換方案」（每方案設定每單位消耗幾福報、換得多少金錢）。
  - 前台只顯示「每種方案最高可兌換多少現金」與「每單位獲得金錢」；玩家選方案後輸入**兌換單位數**（非次數）。
  - 確認後後端自動扣除福報、增加金錢，玩家無感。
- **銀行借貸** `/bank`：借貸系統
  - 取消小/中/大額的固定借款，改為**「依福報額度靈活借款」**機制（操作體驗與換匯所一致）。
  - 前台**完全不顯示任何福報相關資訊**，只顯示「最高可借入金額」與「每單位可借入金錢」。
  - 後台可設定多種「借貸方案」（每方案設定每單位抵押多少福報、可借入多少金錢、每回合產生多少利息）。
  - 玩家選定借貸方案後，輸入**借貸單位數**。系統會即時計算「剩餘可借額度」：
    - **最高可借總單位** ＝ `FLOOR(當前福報 / 每單位抵押福報)`
    - **本次可新增借貸單位** ＝ `MAX(0, 最高可借總單位 - 玩家目前已持有的該方案借款單位數)`
    - 若玩家後續福報下跌，導致「已借單位」超過「當前最高可借總單位」，則無法再新增借款，但原有借款不會被強制結清（會持續扣除利息直到破產或還清）。
  - 每回合主持人按「下一回合」時，系統於後台靜默扣除借款玩家的福報與金錢利息，玩家無法從前台感知。
  - 玩家後續可再以金錢將借款還清。
- **玩家轉帳** `/transfer`：玩家間金錢匯款
  - **不顯示玩家名單**；需輸入完整玩家 ID 才查詢顯示對方資料。
  - 支援 QR 掃碼自動填入 ID。
  - 查詢成功後顯示對方姓名與 ID 卡片，輸入金額與備註後確認轉帳。
- **道具列表**：顯示由關主發放給該玩家的道具。
- **股票市場入口**：點擊進入股市介面。
- **重新整理按鈕**（🔄）：頁面右上角顯眼位置，玩家頁與股市頁皆有
  - 點擊後拉取最新狀態（被關主套快捷、回合結算扣息等被動變化）
  - **節流**：每位玩家在 cooldown 期間只能手動刷新一次（兩個頁面共用同一 cooldown）；秒數讀 `AppSettings.ManualRefreshCooldownSeconds`，預設 60
  - 後端用 atomic SQL 強制節流，繞過前端也擋得住
  - 前端按下後 disabled 並顯示倒數秒數（例：「重新整理（48s）」）
  - 自身操作（買股、轉帳…）的 response 已經自動帶回新狀態，**不算手動刷新、不消耗 cooldown**

> 以上功能模組在「地獄狀態」下全部停用，僅顯示 QR Code。

### 主題與字級偏好
- 玩家可在 `/settings` 切換**深色 / 淺色模式**與**字級**（sm / md / lg / xl）
- 偏好存於 `localStorage`，不上傳後端，下次開頁仍生效
- 後台（`/admin/*`）與看板（`/display/*`）**強制深色 + md 字級**，不跟玩家偏好變動
- 詳細實作見架構文件 §12.3 主題系統

### 效能與暫存機制
- **關主快捷鍵暫存**：關主前台的快捷模組清單，應暫存於前端頁面（如 `SWR` cache 或 `localStorage`），不要每次刷新頁面都從資料庫重新撈取。
- **全域設定快取**：所有固定版面功能（包含大會看板、玩家介面設定），應多利用前端暫存機制，減少向伺服器頻繁請求，提升系統效能。

## 7. 股市介面

### 7.1 顯示（玩家端 — 不顯示圖表）
- **僅顯示商品列表**：股票代碼、名稱、當前價格、漲跌方向（↑ / ↓ + 紅/綠）
- **不顯示曲線圖 / sparkline**（圖表是看板專用 — 玩家要看走勢請看現場大屏看板）
- **股票檔數上限：10 檔**（活動情境刻意限縮，避免玩家認知負荷）
- **持股狀態顯示**：若玩家持有某支股票，該股票卡片以青綠色邊框高亮，並顯示「持股: N 股」。
- **預期賣出利潤**：對有持股的商品，卡片內額外顯示「預期賣出利潤」（當前市價 - 均攤成本）× 持股數，正負值以不同顏色區分。
- **賣出按鈕狀態**：有持股時賣出按鈕高亮（玫瑰紅）且可點擊；無持股則按鈕灰暗且禁用。
- **頁首資產欄**：除「可用資金」外，新增「庫存市值」顯示玩家持股的即時市值合計。

> **設計意圖**：玩家頁手機顯示有限，畫圖會排擠操作按鈕；曲線分析屬於「現場感」內容，集中投放到大屏看板更符合活動情境。看板顯示重點曲線（最多 4 檔）+ 行情總表 sparkline，玩家抬頭看大屏即可。

### 7.2 買賣機制（玩家 ↔ 大會，**無 P2P 撮合**）
- 所有買賣都是「玩家 vs 系統（大會）」，**不存在玩家對玩家的撮合或對手盤**
- 系統視為無限流動性，依當下 `current_price` 即時成交
- **購買方式**：
  - 輸入股票代碼購買
  - 點擊前台顯示中的商品購買
- **價格邏輯**：依呼叫當下的 `current_price` 成交（後台改價是非同步事件）
- **庫存邏輯**（僅指玩家自己的持股，不是市場庫存）：
  - 買進：扣除金錢，玩家持股 +N
  - 賣出：玩家持股 −N，金錢 +N × 當下價
  - 賣出超過持股 → 拒絕

## 8. 活動看板

### 用途
- 在活動現場以投影機／大電視公開展示行情、事件與跑馬燈
- 可獨立於玩家／關主／管理員介面運作，**不需登入**（透過顯示 token 取得授權）
- 全屏設計、無互動、自動刷新

### 顯示內容

#### 畫面佈局與切換機制

**1. 常規模式（遊戲進行中）**：
採三欄式佈局設計（48 / 32 / 20）：
- **重點趨勢區**（左 48%）：包在外框 glass-panel 內。內部 `grid-cols-3 grid-rows-2` **最多 6 檔** featured 商品。每張卡片含 canvas sparkline + 漲跌百分比 + 當前價。後台 `BoardConfig.featured_stock_ids` 可設多個（陣列），看板按陣列前 6 個顯示。
- **行情總表**（中 32%）：顯示所有 `is_visible === true` 的股市商品（代碼、名稱、當前價格、漲跌幅 % + 箭頭）。
- **風雲榜**（右 20%）：每回合 `tickRound` 後 `BoardConfig.last_tick_at` 變動 → fallback poll 拿到新值。常規模式只顯示前 10 名「排名 + 玩家姓名」（**不顯示金額/分數**保持神祕感）。Rank 1-3 加金 / 銀 / 銅獎牌。

**2. 最終結算模式（遊戲結束後）**：
- 當後台（管理員）觸發「最終結算」後，看板畫面會瞬間切換進入霸氣的霸榜模式。
- **隱藏所有股票資訊**：重點趨勢區與大會行情總表將完全退場隱藏。
- **展開全屏排行榜**：大富翁風雲榜會延展至 100% 滿版，並**解鎖顯示所有後台欄位**（包含：排名、姓名、金錢、福份、健康、業力、重生次數、綜合最終分數）。
- **手動滾動檢視**：畫面預設顯示前 10 名，由現場操作員透過滑鼠或觸控版**手動向下滾動**，以揭曉後續名次（不採用自動跑馬燈）。
- 支援與後台相同的**欄位點擊排序**功能，讓大會主持人可以引導現場玩家觀看各項細部數據的排行榜。
- **名次固定原則**：畫面上的「排名」數字永遠綁定於玩家的「綜合最終分數」，當點擊其他欄位（例如：金錢）進行排序時，雖然資料列上下順序會變，但玩家身上背著的「排名數字」絕對不會改變。

> **不顯示換匯所匯率**：匯率屬於玩家視野不可見的隱藏參數，不應在公開看板曝露。

#### 頁首
- 活動 LOGO、活動名稱（例：「開運大富翁 ── 大廳」）、即時時間、日期
- 連線狀態指示燈（🟢 已連線 / 🟡 重連中 / 🔴 斷線）

#### 底部兩列
- **事件**（上列）：來自後台 `Event` 表的預設事件，依排程／優先順序自動輪播
  - 例：「股神就是你！下午茶時段加碼開始」「業力反噬：本時段所有交易手續費 ×2」
- **跑馬燈**（下列）：管理員臨時發布的訊息（即時、可帶 TTL）
  - 例：開幕致詞、突發公告、現場互動指引
  - 與事件分流是因為「事件」屬於遊戲內容（劇情／玩法），「跑馬燈」屬於現場營運

### 事件 vs 跑馬燈

| 項目 | 事件（Event） | 跑馬燈（Marquee） |
|------|---------------|-------------------|
| 來源 | 後台預設 | 管理員臨時輸入 |
| 設定方式 | 大會後台「事件管理」CRUD + 排程 | 大會後台「跑馬燈」單一輸入框 |
| 顯示時機 | 依 `start_at`／`end_at` 自動上下檔；多筆輪播 | 立即生效，可設定自動清除時間 |
| 用途 | 遊戲內劇情、活動節奏（漲跌觸發、加碼時段） | 現場公告、突發訊息、致詞 |
| 顯示位置 | 看板底部「事件」列 | 看板底部「跑馬燈」列 |

### 操作模式
- 由管理員後台產生「活動看板授權連結」（含 display token）
- 看板端僅需開啟此連結，全屏即可運作
- 自動刷新；斷線時自動重連

### 樣式需求
- 16:9 / 4K 解析度優先排版（不出現滾動條）
- 高對比配色，遠距可讀（最小字級 ≥ 24px）
- 漲跌以紅／綠（可後台切換為紅漲綠跌、綠漲紅跌）+ 上下箭頭圖示

## 10. 安全與效能原則

> 詳細實作見架構文件 §13 傳輸加密與資料安全、§14 效能與容量規劃。

### 10.1 安全原則
- **全站 HTTPS**：所有頁面、API、WebSocket 一律走 TLS
- **Token 帶期限**：玩家 QR 與看板顯示連結都含 nonce + 過期時間，外流亦不能無限使用
- **密碼不入庫**：僅存 bcrypt / argon2id 雜湊；登入接口加 rate limit 防暴力嘗試
- **後端是唯一真相**：所有寫入皆由伺服器驗 session 與權限，禁止信任前端傳入的執行者身份
- **稽核可回溯**：所有金錢、道具、設定變動都寫入交易日誌

### 10.2 效能目標（單場 2 小時活動）
| 指標 | 目標 |
|------|------|
| 同時在線玩家 | ≤ 500 人 |
| 大屏幕看板 | 1 ～ 3 台投放 |
| 股票檔數 | ≤ 10 檔 |
| 股市操作峰值 | ≤ 120 req/s |
| 玩家操作 p95 latency | < 300ms |
| 看板資料更新延遲 | **變動推播 < 1 秒**（Realtime）/ fallback 輪詢 60 秒救援漏推 |
| 5xx 錯誤率 | < 0.1% |

### 10.3 設計取捨
- **角色三分（admin / player / captain）互斥**：關主不參與遊戲本身（無四項參數、道具、持股）；同一人若想兼玩，請另建玩家帳號
- **股市為玩家 ↔ 大會（系統）交易**：刻意不做 P2P 撮合與對手盤；系統視為無限流動性，省去訂單簿與撮合引擎的複雜度，配合 2 小時活動節奏
- **回合制由主持人按鈕推進**：每 10 分鐘按一次「下一回合」同時更新股價 + 結算所有借款利息；不依賴 cron（Supabase 免費版不支援），主持人可依現場節奏微調
- **玩家不輪詢、看板靠 Realtime + 60 秒 fallback**：玩家進頁面 / 下拉刷新 / 自身操作 response 才更新狀態；看板靠 Realtime 推播 < 1 秒生效；徹底避開 500 人輪詢洪流
- **股票買賣不鎖股價 row**：以呼叫當下價格成交，避免高併發時所有買賣被序列化；活動情境可接受極小機率的價格不一致
- **跑馬燈 TTL 上限 120 分鐘**（可調 `BoardMarqueeMaxMinutes`）：防止跨場殘留；長期訊息應改用「事件」而非跑馬燈
- **看板看不到匯率與排行榜**：匯率是隱藏資訊；活動進行中**不公開即時排行**（避免玩家偏離劇情），僅終局結算後在看板顯示最終排行榜
- **`/admin` 桌面優先（≥1280px），`/captain*` 手機優先**：管理員多在桌機操作大表格，關主在現場拿手機掃碼

### 10.4 高併發場景防禦（如：500 人同時抽卡）
針對實體活動常見的「主持人喊倒數，全場同時操作」的瞬間流量（Spike），系統具備以下防禦機制以避免資料庫癱瘓：
1. **啟用連線池 (Connection Pooling)**：伺服器不使用直連，而是透過交易模式的連線池（如 Supabase PgBouncer，Port 6543）將瞬間湧入的請求優雅排隊，避免資料庫連線數耗盡。
2. **命格題庫快取**：抽卡所需的「命格範本」清單暫存於伺服器記憶體中，不重複向資料庫查詢，伺服器僅負責亂數計算並將最終結果單次寫入資料庫。
3. **資料庫寫入防呆 (Idempotency)**：寫入玩家初始參數時，更新語句強制加上 `WHERE destiny_name IS NULL` 條件。防止玩家在網路延遲時因連續點擊按鈕而導致重複寫入或重複領取初始資源。
4. **前端動畫視覺分散 (Jitter)**：玩家按下「求籤」時，前端先播放 2~3 秒的搖籤動畫，並在背後加入 100~1500 毫秒的隨機延遲再發送 API 請求。此舉能將原本擠在 1 秒內的 500 個請求，在使用者無感的情況下打散到 3 秒內完成，將瞬間海嘯化解為無害的小波浪。
