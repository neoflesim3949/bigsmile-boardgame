| 1.6 | **銀行借貸**（玩家自行借貸生成合約）| 後台設方案、合約化還款 | 無 | ⏸️ **未討論** |

## 設計取向核心差異

### A. 自動化 vs 手動化

| 面向 | Neo V2 | 體系版 |
|------|--------|--------|
| 玩家買賣股票 | 玩家自助 | **關主代操** |
| 重生 | 自動套重生初始值 | **GM 手動配** |
| 業力影響 | 每回合自動 | **不做** |
| 借款利息 | 每回合自動扣 | **無借貸機制** |

→ 體系版**重操作門檻、輕系統自動**，更接近「桌遊以人為主、系統輔助」。

### B. 命運機制 vs 角色機制

| Neo V2 | 體系版 |
|--------|--------|
| 抽命格（隨機 + ratio）| 選角色 |

→ Neo 偏「命運論」（你抽到什麼是天意 / 機運），體系版偏「角色扮演」（你選擇你要的）。

### C. 經濟系統複雜度

| Neo V2 | 體系版 |
|--------|--------|
| 股市 + 換匯 + 借貸 + 業力扣分 + 強制平倉 | 沒有經濟系統（只有計分）|

→ Neo 有完整虛擬經濟、體系版專注核心積分玩法。

## 跨版本實作可行性建議

### 體系版要實作 Neo 的功能，需要的工程量

| 功能 | 體系版實作難度 | 建議優先序 |
|------|--------------|-----------|
| 需求 1（負值提示）| 🟢 低（純 UI 邏輯）| **必做** |
| 需求 2（計分權重）| 🟢 低（後端公式 + admin UI 可調）| **必做** |
| 需求 3（單維上限）| 🟢 低（同需求 1）| **必做** |
| 換匯所 | 🟡 中（後端方案表 + 玩家 UI）| 待 PM 決議 |
| 銀行借貸 | 🟠 中高（合約模式 + 利息結算）| 待 PM 決議 |
| 命格抽卡 | 🟠 中高（演算法 + 範本管理）| 體系版「角色」已部分覆蓋、不必補 |
| 業力影響 | — | 5/5 已決定不做 |
| 玩家自助股市 | — | 5/5 已決定不做 |

### Neo V2 要簡化向體系版靠攏，需要的工程量

如果 Neo 想關掉某些功能讓玩法接近體系：

| Neo 功能 | 關閉方式 |
|----------|---------|
| 業力影響 | admin 把所有 KarmaBand 的 delta 設 0 |
| 換匯所 | 不創 ExchangeOption row（玩家點 /exchange 看到空清單）|
| 銀行借貸 | 不創 BankLoanOption row |
| 玩家自助股市 | 把所有 Stock `is_visible=false` 並 `is_sellable=false`（或單純不告訴玩家股市路徑）|
| 抽命格 | `AppSettings.CardDrawMode='false'`（middleware 不導 /onboarding） |

→ Neo V2 設計上即支援「關閉部分功能」，可彈性配置。

## 結論

**兩版本定位明顯不同**：

- **Neo V2**：**功能完整、自動化高、虛擬經濟深、命格機制**。適合「玩家自主、系統做事多」的活動。
- **體系版**：**功能精簡、人為操作多、純積分玩法、角色機制**。適合「以人為主、系統當記分板」的活動。

**5/5 會議的決議走向**：體系版**不擴張為 Neo V2**，而是補足 3 個必要需求（負值提示 / 計分權重 / 單維上限）後維持精簡定位。

**未來潛在合作**：若體系版未來要加經濟系統（換匯 / 借貸），可參考 Neo V2 的 schema 與規範。
